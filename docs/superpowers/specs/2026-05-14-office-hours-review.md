## 审视方法

已有完整设计文档（2026-05-14-poe2wiki-design.md），本次 office-hours 通过前提挑战和三方案对比来纵深审视执行策略。

## 前提挑战结果

三个前提全部通过用户确认：

1. **PoE2 玩家需要独立攻略站** — 用户确认。竞争格局：poe2db.tw 做数据，PoB 做计算，但没有一个把两者集成+推荐系统的中文产品。
2. **集成 PoB 引擎是正确的** — 用户确认。准确性优先于开发速度。
3. **一步到位做全部功能** — 用户确认，但经过替代方案对比后，调整为"数据MVP→计算器优先→逐步补全"的垂直切片策略。

## 方案选择

**选择：方案C修正版 — 计算器优先，数据MVP支撑**

执行顺序：
```
数据抓取(poe2db.tw) → 数据MVP入库 → DPS计算器+推荐 → 补全数据库 → 攻略+社区
```

与原方案A（全栈瀑布流）的关键差异：
- 数据管理系统（后台CRUD）不是第一优先级，先用脚本灌数据
- DPS计算器作为首个上线的核心功能，承载差异化价值
- 攻略和社区后置，但数据模型预留

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 数据抓取不稳定 | 先抓取静态快照，后续增量同步 |
| PoB引擎集成比预期复杂 | 预留简化计算器作为 fallback |
| 计算器上线后无内容吸引留存 | 计算器分享链接自带传播属性 |
